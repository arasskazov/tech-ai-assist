Структурированная бизнес-аналитическая записка обеспечения функциональной устойчивости ОКИИ

---

# **1. Введение**
При эксплуатации интегрированных автоматизированных и информационных систем состоящих из совокупности объектов критической информационной инрфраструктуры показатеи надежности, отказоустойчивости, безопасности, применяемые в отдельности, не отвечают совлеменным требованиям по обеспечению непрерывности выполнения ключевых бизнес-процессов. Расчет и подтверждение проектных требований при развитии автоматизированных систем представляет собой крайне трудоемную задачу, при этом не учитываются различия в степени важности ключевых, основных и обеспечивающих процессов, требуется более широкое применение риск-ориентированного подхода.

Для решения компромиссной задачи обеспечения приемлемого уровня функционирования ОКИИ, обеспечения развития функциональности, эксплуатации в условиях внешних дестабилизирующих факторов, различных рисков, предлагается определение функциональной устойчивости, создание инфраструктуры непрерывного мониторинга фактических параметров (метрик) функциональной устойчивости Объектов КИИ (ИС) на основе стандартизированных процедур. Текущая практика управления ОКИИ не позволяет объективно подтверждать реальный уровень устойчивости и учитывать его в экономике и регулировании.

Предлагается внедрить **систему подтверждаемой функциональной устойчивости**, в которой состояние инфраструктуры непрерывно измеряется, подтверждается на основе телеметрии и тестов, проходит независимый аудит и трансформируется в управляемый показатель (индекс устойчивости).

Это позволит:

* превратить устойчивость в **измеряемый актив**,
* обеспечить **объективное подтверждение состояния ОКИИ**,
* связать устойчивость с **экономическими и регуляторными последствиями** (страхование, финансирование, требования регуляторов, доступ к рынку КИИ).

# **2. Краткое описание**

## 2.1. Текущая ситуация (Current State)

| Область    | Состояние                        | Ограничение                       |
| ---------- | -------------------------------- | --------------------------------- |
| Нормативка | Требования к устойчивости заданы | Нет интегрированного измеримого показателя |
| Контроль   | Аудиты, проверки                 | Периодичность, субъективность, в основном ИБ     |
| Метрики    | RTO, RPO, SLA и др.              | Разрознены, не агрегируются       |
| Данные     | Частично собираются              | Нет доверенной среды              |
| Экономика  | Не связана с устойчивостью       | Нет стимулов для инвестиций       |

---

## 2.2. Целевое состояние (Desired State)

Формируется интегрированный контур измеримых показателей:

| Этап                      | Суть                          |
| ------------------------- | ----------------------------- |
| Функционирование          | Реальная работа системы       |
| Телеметрия / тесты        | Объективные данные            |
| Подтверждение             | Формализованные подтверждения |
| Аудит                     | Независимая верификация       |
| Индекс устойчивости       | Интегральная оценка           |
| Экономика / регулирование | Последствия                   |

## 2.3. Концепция

### Основной принцип:

> **Функциональная устойчивость** — свойство информационных систем и ИТ-инфраструктуры объектов критической информационной инфраструктуры обеспечивать выполнение критически важных функций (процессов) и услуг путем сохранения или оперативного восстановления заданных параметров качества (доступности, производительности, целостности) при возникновении любых событий, отклоняющих функционирование от нормального режима, включая сбои и отказы оборудования, ошибки программного обеспечения, ошибочные или преднамеренные действия персонала и третьих лиц, кибератаки, изменения нагрузки и внешней среды, а также адаптироваться к изменениям для минимизации негативных последствий.

### Методологическое ядро:

| Компонент        | Содержание                                    |
| ---------------- | --------------------------------------------- |
| Объектная модель | Типовой объект → функции → риски              |
| Метрики          | Набор измеримых параметров                    |
| Подтверждение    | Формализованные свидетельства (данные, тесты) |
| Индекс           | Агрегированная оценка                         |
| Доверие          | Контроль целостности данных                   |
| Аудит            | Независимое подтверждение                     |

### Интерпретация метрик

* Индекс устойчивости ниже порогового - граничения по эксплуатации КИИ
* Величина индекса влияет на величину штрафа при инцеденте
* Величина индекса влияет на рейтинг, рейтинг влияет на страховые платежи


### Применение

| Область       | Эффект                 |
| ------------- | ---------------------- |
| Страхование   | Дифференциация тарифов |
| Инвестиции    | Оценка риска           |
| Регулирование | Гибкие требования      |
| Эксплуатация  | Допуск/органичения КИИ |
| Контракты     | SLA нового поколения   |

### Уровни метрик

| Уровень | Тип метрик                  | Пример                      |
| ------ | --------------------------- | --------------------------- |
| 1      | Бизнес-метрики устойчивости | Потери, недоступность услуг |
| 2      | Сервисные (SLA/SLO)         | Доступность 99.95%          |
| 3      | Технические (SLI)           | latency, error rate         |
| 4      | Инфраструктурные            | CPU, сеть, отказ узла       |
| 5      | Организационные             | компетенции, процессы       |

## Методика определения метрик

### Процесс задания метрик

 1    Определение критичных функций         
 2    Выделение критичной архитектуры       
 3    Определение рисков                    
 4    Выбор метрик                          
 5    Установка целевых значений            
 6    Определение источников данных         
 7    Определение правил фиксации инцидентов
 8    Введение процедур контроля            

### Типы целевых показателей

| Тип                | Пример                 |
| ------------------ | -------------------------- |
| Доступность        | uptime                     |
| Восстановление     | RTO, MTTR                  |
| Деградация         | доля операций с ухудшением |
| Производительность | latency, throughput        |
| Надежность         | MTBF                       |
| Управляемость      | скорость реакции           |

## Механизм подтверждения

### Источники подтверждения

| Источник        | Тип данных        |
| --------------- | ----------------- |
| Телеметрия      | автоматическая    |
| Тесты           | активные проверки |
| Журналы         | события           |

### Требования к подтверждению

## Интегральная модель оценки

### Структура

| Компонент      | Вес | Пример    |
| -------------- | --- | --------- |
| Доступность    | 30% | SLA       |
| Восстановление | 25% | MTTR      |
| Обнаружение    | 15% | MTTD      |
| Защищённость   | 15% | Инциденты |
| Управляемость  | 15% | Процедуры |

### Формула

```
Индекс = Σ (Метрика_i × Вес_i × Коэффициент подтверждения)
```

Где:

* **Коэффициент подтверждения** отражает качество подтверждения:

  * 1.0 — подтверждено телеметрией
  * 0.7 — тестами
  * 0.5 — документально

## Обеспечение доверия к данным

**Данные могут быть:**

* неполными,
* искаженными,
* нерепрезентативными.

**Решение:**

| Механизм               | Назначение           |
| ---------------------- | -------------------- |
| ПАК (ДПАК)             | Сбор и защита данных |
| Контроль целостности   | Исключение подмены   |
| Непрерывный мониторинг | Актуальность         |
| Трассируемость         | Проверяемость        |

# 4. Пилотный проект

Задача пилотного проекта - подтверждение применимости показателей на ОКИИ Росатома, ТЭК

* Объекты оценки — информационные системы и ИТ-инфраструктура объектов КИИ.

* Что измеряем:
  - Критическая функциональность — способность поддерживать заданные бизнес-процессы даже в условиях сбоев.
  - Восстанавливаемость — метрики RTO, RPO, MTTR, MTTD.
  - Доступность — процент времени готовности сервисов.
  - Целостность и полнота данных — контроль искажений, потерь телеметрии.
  - Управляемость и готовность — полнота планов, регулярность учений, успешность изменений.
  - Внешняя устойчивость — зависимость от поставщиков, санкционные риски, географическая распределённость.
  - Базовая работоспособность оборудования — состояние БП, RAID, температуры.
  - Интероперабельность — поддержка протоколов управления, автоматизации, совместимость с российским ПО.

* Диапазон воздействий (что учитываем):
  - Технические сбои (отказы железа, каналов, ПО).
  - Кибератаки (включая APT, ransomware).
  - Отказы поставщиков и внешних сервисов (уход с рынка, санкции).
  - Природные и техногенные катастрофы (пожар, отключение энергии).
  - Регуляторные изменения (только в части их влияния на ИТ-ландшафт).

## Архитектура решения (укрупнённо)

| Слой          | Компоненты                |
| ------------- | ------------------------- |
| Данные        | Телеметрия, тесты         |
| Обработка     | Расчёт метрик             |
| Подтверждение | Формирование свидетельств |
| Хранилище     | Реестр подтверждений      |
| Аналитика     | Индекс                    |
| Представление | Витрины                   |

## Ближайшие шаги

1. Утвердить концепцию **подтверждаемой устойчивости** как базовую.
2. Запустить пилот:

   * 2–3 объекта (ТЭК, Росатом)
   * полный цикл: данные → подтверждение → аудит → индекс
3. Создать:

   * реестр метрик,
   * методику расчёта индекса,
   * модель подтверждений.
4. Интегрировать результат в:

   * страховую модель,
   * допуск к эксплуатации КИИ,
   * регуляторные требования.
